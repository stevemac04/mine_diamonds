Reinforcement Learning Minecraft Agent

## Introduction

Reinforcement learning has achieved strong performance in structured environments such as board games, robotics simulations, and arcade games, where agents operate within clearly defined objectives and relatively constrained environments. Open-world games introduce a significantly greater challenge because agents must operate in large environments with sparse rewards, incomplete information, and long sequences of dependent actions. Minecraft is one of the most popular and well-known examples of an open-world sandbox game, and we decided to use it as a benchmark for our artificial intelligence research and learning because it combines exploration, resource management, crafting, and long-term planning into a single environment.

In Minecraft, players spawn into a procedurally generated world where they must gather resources in order to progress. Early gameplay typically begins with locating trees and collecting wood by punching logs. Wood can then be converted into planks and used to craft basic tools, such as pickaxes, axes, and shovels. These tools allow players to mine certain resources more efficiently, like stone, wood, and dirt, which can eventually be used to craft stronger tools. The strongest tool, at least until a few years ago, was diamonds. To eventually obtain these diamonds, players must continue progressing through multiple tiers of tools and stages including collecting iron, smelting materials, crafting advanced tools, and navigating underground cave systems. While these tasks are relatively intuitive for human players, they present major challenges for reinforcement learning agents because success requires completing multiple prerequisite objectives in the correct order while navigating unpredictable and randomly generated environments. In other words, a player can not simply spawn in a world and go get diamonds instantly.

Our original goal was to design a reinforcement learning agent capable of spawning into a randomly generated Minecraft world and autonomously gathering the necessary materials required to eventually mine diamonds. We planned to train this agent through reward shaping, where intermediate rewards would be given for completing smaller tasks such as locating trees, mining wood, collecting resources, crafting basic items, and eventually gathering stronger materials required for diamond progression. However, as development progressed, we discovered that directly training an agent in a standard Minecraft world introduced several major challenges. Infinite worlds, complex terrain, and long sequences of dependent actions made full diamond progression unrealistic within the scope of this project. As a result, we narrowed our final objective to the more attainable early-game task of training an agent capable of reliably identifying trees, mining wood, collecting dropped resources, and verifying successful inventory updates. This revised objective allowed us to focus on building a functional reinforcement learning framework while still addressing the challenges of exploration, visual recognition, action execution, and sequential task completion of an open-world environment.

## Methodology

To begin, our reinforcement learning model would need to have the ability to make inputs into Minecraft, and understand the visuals of the environment that surrounds it. We allowed the agent to train and be evaluated directly in Minecraft Java Edition running on our computer. Instead of using a simulated environment, the agent interacted with the live game client through a screen capture of the window that it had direct access to and synthetic keyboard and mouse inputs to control the player. Quickly, the open-world environment of Minecraft started to display how difficult our task would be. Minecraft has 12 different wood variants, all having their own unique texture, color, and name when collected.

These different wood types would need to be identifiable from the over 800 different block types that appear in game. Training our agent to decipher these wood types from other blocks would require some sort of internal computer vision model by itself. To simplify this, as we figured out we would have to deal with many of the different obstacles presented by the open world of Minecraft, we created a texture pack that made all of these textures uniformly black; something far easier for the agent to identify when searching for a tree to chop down.

These consistent textures for the goal of wood and the agent having access to a virtual controller for the player is great, but how was the reinforcement learning actually going to take place? Whenever the agent sends an input through to the keyboard/mouse it will receive the current frame of the game as a visual input. From these inputs, it will learn by connecting what it takes in visually to what follows afterwards. This is where a clear issue appeared. With mining wood as the goal, what could the model even be rewarded for? Reward shaping proved to be a clear issue right from the start because of this. Whenever an agent was spawned in, it would be able to run around, jump, look any which way, punch, but in a 3-dimensional world with tons of blocks to look at and experiment with, almost every iteration would ultimately be a "failure" in the sense of never being able to actually obtain wood.

To solve this problem, reward shaping was necessary. Instead of waiting for a stroke of absolute luck where the agent randomly made inputs that resulted in a tree being broken down, we added smaller steps along the way that we could reward behaviors for. The first step we introduced was rewarding visible evidence of logs in the frame. Continuing from this, rewards were increased as the agent moved closer and the logs became visually larger. Facing the logs at the center region of the screen came with additional awards. Finally, we incentivized the agent to chop down (attack) the tree while aimed. The sparse success reward for actually acquiring a log in the hotbar was still kept as the main objective, but these intermediate rewards gave the policy enough feedback to begin learning how it could reach that objective.

To help make these intermediate rewards, we introduced two key visual regions that would help the agent understand when it is aimed at wood and when it acquired wood. The first was a center "fovea" region where the agent was aimed; the second was the first hotbar slot where the log would end up. At the start of each episode, we captured a baseline image of an empty slot 1 and then measured visual difference from that baseline during runtime. When slot 1 changed in a stable way consistent with collecting an item, that event was treated as log acquisition and triggered the sparse success reward.

Even after these reward and vision changes, we observed that apparent progress could be misleading, so we evaluated each run with stricter acquisition checks and compared multiple training configurations rather than relying on a single headline metric.

## Results

We evaluated performance using per-episode outcomes and training diagnostics across multiple real-Minecraft runs. Our key metric was verified log acquisition per episode, with additional diagnostics for aiming behavior, log visibility, and policy-update stability. Each episode was capped at 60 seconds of in-game time, but the realized episode lengths varied based on whether the agent reached terminal conditions early or timed out.

In an early no-kill-reset experiment, we ran 40 episodes over approximately 14.5 minutes of wall-clock time (2,018 total agent timesteps, mean episode length 50.5 steps). The agent showed strong short-run behavior signals, including high aim and frequent detector-triggered acquisitions. However, after tightening the acquisition criteria (requiring stronger and more stable slot-change evidence), we found that some of these events were false positives. Because of this, we do not treat those early high rates as final task success.

In our longer superflat run with consistent reset logic (`y=0` and periodic relocation), we ran 104 episodes over approximately 162.6 minutes (36,339 total agent timesteps, mean episode length 349.4 steps). The agent produced the most stable behavior over time and repeatedly reached tree-related states. This run gave us the clearest evidence that policy updates were learning useful intermediate behaviors rather than only random movement. At the same time, strict success validation remained the limiting factor: learning to see and approach logs was easier than consistently completing the full mine-and-confirm sequence.

A strict bidirectional-exploration retry run further highlighted this gap. In that trial, we observed 4 long episodes over approximately 41.1 minutes (9,454 total agent timesteps, mean episode length 2,363.5 steps). The policy accumulated substantial shaped reward and showed tree-facing behavior in several episodes, but still failed to produce validated end-to-end acquisitions in that trial. This confirms that improved exploration alone does not guarantee completion of the final objective.

Overall, our parameter search suggests that environment and reset design were as important as model tuning. We made measurable progress on exploration, targeting, and stability, but robust, strictly validated log acquisition remained the hardest part of the pipeline.

## Conclusions

This project showed that reinforcement learning in live Minecraft is practical to implement but difficult to stabilize. Our biggest lesson was that environment control, state reliability, and reward design can matter as much as the learning algorithm itself.

We also learned that intermediate improvements can overstate true progress if evaluation is not strict. In our case, stronger validation rules exposed false positives and gave a more honest picture of system capability. That made the results less flashy, but far more defensible.

From a systems perspective, we successfully built and tested a full real-time RL loop: live visual input, synthetic control, episode resets, reward shaping, and training diagnostics. From a task perspective, the agent learned parts of the wood-collection sequence, but consistent verified completion is still unresolved.

The main outcome of this work is a functional real-Minecraft RL framework and a clear experimental record of what worked, what failed, and what must be improved next. Future work should focus on stronger success detection, better mining/collection commitment behavior, and curriculum designs that bridge the gap between "finding wood" and "reliably acquiring wood."
